Note that it would be useful to provide some context for these opinion statements
given that most users of cc-cmt are seeing an e-mail response to relatively old
question, requiring some work to establish the necessary context. Perhaps the
question could be copied (if short enough) or a link could be provided to the
question in the cc-cmt repository.

The requirement (FAU_GEN.2) quite simply indicates that each auditable event must
be associated with the user that caused the event. I do not think there is any
real implication that a user identity cannot be changed, so I don't think any word
change should be necessary. Without the context of the original question, the last
sentence of the NIB opinion seems too strict (i.e., generation of identity change
records is not generally the only solution). However the original question is
specifically asking about changes to identity.

Original question:

"FAU_GEN.2 requires that the TSF be able to associate each auditable event with
the identity of the user that caused
the event. Does this mean that even a process with relevant TSF privileges can not
change its own user ID (nor any
other user/audit related ID)? And the fact that earlier records in the audit trail
could show any changes to the user ID
of the process does not help?"

NIAP Interpretations Board wrote:

> "...
>
> A very strict reading of the requirement could lead one to believe that no
> change of user identity is permitted. Perhaps some rewording is in order. The
> NIB's opinion is that the requirement means that the TSF shall provide
> facilities so that the identity of the user may be determined. The TSF must
> therefore generate audit records that show who changed their identity and to
> what they changed it."
